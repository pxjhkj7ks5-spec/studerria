Правила для воркспейсу KMA

Ці правила застосовуються тільки в `C:\Users\andri\OneDrive\Desktop\proj\kma`.

- Після будь-яких змін піднімай версію на `+0.0.01` через `version.js patch` у форматі на кшталт `0.7.00 -> 0.7.01`.
- Дотримуйся стилю `macOS Tahoe / Apple Liquid Glass / iOS 26`.
- Після кожного оновлення оновлюй `changelog.json` на основі нових git-комітів.
- Формат `changelog.json`:
  - Для звичайних версій (`x.y.zz`, де `zz != 00`) використовуй короткі загальні формулювання без технічних деталей і без опису внутрішніх адмінських змін.
  - Для круглих релізів (`x.y.00`) пиши 2-4 пункти з поясненням змін, що реально впливають на користувацький досвід (розклад, журнал, дедлайни, teamwork, My Day, профіль, підтримка).
  - Внутрішні адмінські/інфраструктурні зміни не деталізуй у changelog; описуй їх лише узагальнено як покращення стабільності/якості.
  - Ротуй нейтральні шаблони на кшталт: `Мінорні оновлення сайту для покращення роботи.`, `Локальні покращення стабільності та зручності використання.`, `Поточні доопрацювання для кращого щоденного досвіду.`
- Відображай `changelog.json` через модалку, яка відкривається кнопкою `Changelog` у футері.
- Актуальний сценарій деплою та оновлення: `Docker Compose` із `docker/local`; `Cloud Run` більше не є робочим цільовим середовищем.
- Для звичайного серверного оновлення використовуй `git pull --rebase` у корені репозиторію, потім `docker compose up -d` у `docker/local`; `--build` використовуй тільки коли змінені Dockerfile/залежності/збірка.
- Після кожного завершеного оновлення в відповіді обов'язково додавай короткий блок команд для запуску оновлення на сервері.
- Локальні тести запускай за потреби на власний розсуд для ризикових або інфраструктурних змін; не чекай окремого запиту, якщо перевірка справді потрібна.
- `docker compose down -v` використовуй тільки для повного перестворення Postgres volume або повторного імпорту SQL-дампу, а не для звичайного апдейту.

## Parallel Work In One Repository

У цьому репозиторії може бути кілька незалежних проєктів. Якщо користувач прямо просить працювати паралельно, дозволено вести до 3 паралельних потоків роботи або сабагентів тільки за таких умов:

- Кожен потік має свій чіткий scope і свій каталог проєкту.
- Кожен потік має непересічний `write set`: не можна одночасно редагувати ті самі файли або спільні конфіги без окремого явного плану.
- Якщо зміни зачіпають `shared`-код, root-конфіги, спільні залежності, CI, Docker, env templates, спільну UI-бібліотеку, базу даних або міграції, це не вважається незалежною задачею і має виконуватись послідовно або окремо координуватись.
- Перед стартом треба явно визначити `Project A`, `Project B`, `Project C` і закріпити за кожним його каталоги або файли.

### Rules

- Спочатку коротко розбий задачу на незалежні частини.
- Один агент або потік відповідає тільки за один проєкт або одну ізольовану зону.
- Не переносити зміни між проєктами без прямої потреби.
- Не чіпати незавершені локальні зміни в інших частинах монорепо.
- Якщо виявлено конфлікт по файлах, залежностях або інтеграційній логіці між потоками, зупинити паралельну роботу і перейти до послідовного виконання.
- Головний агент відповідає за інтеграцію, фінальну перевірку, коміт і підсумок.
- У фінальній відповіді звітувати окремо по кожному проєкту: що змінено, де змінено, як перевірено.

### Preferred Execution Model

1. Виділити до 3 незалежних підзадач.
2. Для кожної підзадачі закріпити конкретні каталоги або файли.
3. Паралелізувати тільки незалежні частини.
4. Після завершення звести результат і перевірити інтеграційні ризики.

### Safety Rules For Monorepo

Не запускати паралельно, якщо:

- кілька задач змінюють один і той самий shared module;
- кілька задач змінюють root-level залежності або збірку;
- кілька задач змінюють міграції, схему БД або спільні типи;
- незрозуміло, де проходить межа відповідальності між проєктами.

У таких випадках треба або:

- спочатку винести shared-зміни в окремий крок;
- або виконувати задачі послідовно.
